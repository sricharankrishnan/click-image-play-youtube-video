Introduction. What and Why Jekyll Lava Boilerplate?

Jekyll Lava Boiler Plate is a small project built to help developers by reducing that extra time in building a Jekyll Project Environment. It comes pre-built with certain features that can be useful for production level sites.

This boiler plate project is currently being served with Jekyll Gem 3.6.3. Lets get you started on how you can use this.

How To Install/Download Jekyll Lava Boilerplate?

Before downloading this boilerplate, make sure that you have ruby and jekyll pre-installed in your system. Here is a link from the Jekyll Source Page itself: https://jekyllrb.com/docs/installation/.

Then, Jekyll Lava Boiler Plate can be downloaded either directly by using the Download Button at the Github Project Repo or you could do a git clone.

git clone https://github.com/sricharankrishnan/jekyll-lava-boilerplate.git [your-project-name]

Features

Jekyll Lava Boiler Plate comes with some simple yet effective features that can assist you in your building process. These features are based on the documentation from the original Jekyll Website. The features and ideas involved are:

Layout Directory (./_layouts): The layouts directory will be the central point from which all necessary details per page will be served from. You can have more than one layout page and this can be changed in the individual page's layout variable.

SEO Tag(s) being served with Page Variables The title tag and meta description tags have their respective content that would be served from a page's title and metaDescription variables. You could go ahead and replicate this style of populating SEO details per page like so. Check: ./_layouts directory

relBase Feature - Handling urls for .css/.js files etc.. The rel-base.html document is being served from the Includes directory (./_includes). This is a really cool feature that was picked up from https://ricostacruz.com/til/relative-paths-in-jekyll. This helps in ensuring that all assets that need to be locally served like .css and .js files are propogated based on the required path. Check: ./_layouts directory

404 Page Not Found Handler Need to have a fallback for a Page Not Found (404)? No problem. A conditional if statement detects whether a page is being served with the 404.html and if so, shall automatically refresh to the home page post 30 seconds. You can adjust this redirect timer in the respective meta content value. Check: ./_layouts directory

Favicon(s) Favicons, just like the title and meta desciption tags, may be served from the respective page. This style comes in handy if you have special promotions or indiuvdual sport pages. You could decide to just keep one favicon by removing the href details and place your respective favicon url. Check: ./_layouts directory

Stylesheets Wondering why I have kept conditional if based rendering for this? Funny right? Not necessarily and here is the reason why. The idea is that every page may not only have its own set of styles but also styles that are commonly applicable. For example, if you are on home page, your stylesheet draws a font-file style and some common variables with the Sass @import feature. This together with the indvidual styling may ensure that your page has styles applicable to it only and no other page. Check: ./_layouts directory Check: ./_sass directory Check: ./css directory

page.bodyClass Is useful for propagating the stylesheet with the conditional rendering and also, placing the required css class value for a body tag on that page. Check: ./_layouts directory Check: ./index.html

Html Element Div.outerBorder and the Curly Braces
content inside it div.outerBorder contains the required code for picking up with content from every page that uses the default layout. Javascript Tags at the end help with the neccessary javascript code placement which you may write depending upon your project requirement. Check: ./_layouts directory

Sass Directory (./_sass): This is the directory from which all imports being made by stylesheets will have details imported from. A useful place for establishing details like Website Fonts, Sass Variables etc. Using @import "file-name" will ensure that you get/import the details from the necssary file from this directory. Checkout the ./css/styles.scss file to get an idea of what is being done to achieve this. You should not use Front Matter Initialization in files placed in the ./_sass directory. What is a Front Matter? Check out this link Check: ./_sass directory Check: ./_css directory

Includes Directory (./_includes): This directory is used for housing files that could perhaps be served commonly across the entire website. A good place for keep all HTML relevant details for your website Navigation and Footer Sections. Would recommend checking out https://jekyllrb.com/docs/includes/ for more information on how a includes file would be placed and imported into your respective web pages.

Data Directory (./_data): This directory is useful for placing your custom data that could be served onto your webpage through Liquid Templating. Think about not having to go through heaps of HTML codes but you can make changes to your webpages in your localhost just by accessing your respective data file. You could also use the ./_data Directory for listing out details that could perhaps be placed inside an unordered or ordererd list tag. If you would like to see more examples, check out https://jekyllrb.com/docs/datafiles/ for more information.

Minify HTML HTML files can be minified in Jekyll. A file by the name of compress.html is kept in the ./_layouts directory. The required Front Matter is placed at the top of ./_layouts/default.html file. This helps in compressing/minifying the html page being served. During development process, you could remove the top level front matter in your respective layout file(s) and place it back again to compress the final html file being rendered. Make sure not to delete the compress.html file If you would like to have more information about this plugin, check out https://alligator.io/jekyll/compress-html/ and https://github.com/penibelst/jekyll-compress-html. Check: ./_layout directory Check: ./_layout/compress.html file

Minifed CSS This jekyll build handles the default Sass preprocessor for writing your respective stylesheet details. Necessary configuration is in place. Check: Inspect Browser and check the css file Check: ./_config.yml file
